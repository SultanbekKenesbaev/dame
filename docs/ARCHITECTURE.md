# Архитектура DailyGate

## Поток допуска

1. Windows загружает managed auto-logon профиль, а Shell Launcher запускает `DailyGate.Client` вместо Explorer.
2. Клиент получает статус через named pipe. Служба вычисляет `workday_id` по локальной конфигурации, сверенной с сервером.
3. При наличии подписанного completion receipt служба сообщает `Unlocked`, клиент запускает Explorer и скрывается.
4. Иначе сотрудник входит в клиент. Онлайн-проверка обновляет hardware-bound offline verifier и семидневную lease.
5. Служба выдаёт только тест с корректной ECDSA-подписью. Правильные ответы в payload отсутствуют.
6. Перед сетевой отправкой ответы транзакционно попадают в локальный SQLite WAL. При отсутствии сети рабочий стол открывается, а очередь синхронизируется позже.
7. В 03:50/03:59 скрытый клиент показывает предупреждение. При изменении `workday_id` служба вызывает `WTSLogoffSession`.

## Границы доверия

- WPF-клиент работает как стандартный kiosk-пользователь и не читает ключ устройства или SQLite напрямую.
- Windows-служба работает как `LocalSystem`, владеет ключом, DPAPI-данными, временем и logoff.
- Named pipe bootstrap-ACL разрешает локальную связь аутентифицированному сеансу, после чего служба привязывает канал к SID первого стандартного kiosk-профиля; администраторы и другие SID отклоняются.
- Устройство подписывает каждый API-запрос ECDSA P-256. Каноническая строка содержит метод, path/query, timestamp, nonce и SHA-256 тела.
- Сервер отклоняет timestamp старше пяти минут и повторный nonce.
- Сервер ECDSA-подписывает тесты и completion receipts. Публичный ключ фиксируется при enrollment.
- Админка использует same-origin HTTPS, SameSite=Strict HttpOnly cookie и короткую JWT-сессию.

## Данные

PostgreSQL хранит сотрудников, группы, устройства, банки/вопросы, правила, ежедневные экземпляры, ответы, emergency unlocks, device events и append-only audit events. Уникальные индексы обеспечивают один тест и одну отправку на сотрудника/рабочий день.

Каждая отправка сохраняет неизменяемый снимок имени устройства и версий клиента/службы. Поэтому перепривязка ПК не переписывает историческую аналитику. При перепривязке следующая синхронизация очищает локальные тесты, очередь и receipts прежнего сотрудника.

Будущие тесты создаются rolling-window worker-ом на семь дней. Snapshot JSON и порядок вариантов фиксируются при выпуске; изменение банка не переписывает выпущенные экземпляры.

## Отказоустойчивость

- API и worker используют один контейнерный образ, но запускаются отдельными процессами.
- API применяет миграции и bootstrap; worker запускается после healthy API.
- PostgreSQL находится во внутренней Docker-сети и не публикует порт.
- Signing key хранится в отдельном volume и монтируется API/worker.
- Device submissions идемпотентны по `IdempotencyKey`; повторная доставка возвращает исходный receipt.
- При недоверенной подписи, clock rollback или повреждённом кэше служба блокирует допуск и отправляет security event.

## Ограничения первой версии

- Одна организация/заведение.
- Персональная привязка одного активного устройства к одному сотруднику.
- Только single-choice и multiple-choice без оценки правильности.
- Нет email/Telegram-уведомлений.
- Нет автоматического удаления истории.
